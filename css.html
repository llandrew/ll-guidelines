Syntax and Formatting

  • Use soft tabs (2 spaces) to guarentee the file renders the same on all systems.
  • Keep each selector on its own line when grouping selectors unless it makes sense otherwise.
  • Always put a space between the opening brace and the last selector in the call.
  • Always put the closing brace on its own line.
  • Make sure to have a space after the colon for a declaration.
  • Keep each declaration on its own line.
  • Always put a semicolon at the end of the declaration to help avoid any errors.
  • Include a space after each comma in each declaration except for rgb(), rgba(), hsl(), hsla(), and rect().
  • Prefix all decimals smaller than 1 with a 0 (e.g. 0.5rem).
  • Hex values can be all upper or all lower case, but must stay consistent with the rest of the file.
  • Make sure to use quotes when using an attribute selector (e.g. [type="text"]).
  • If a value is 0, it does not need a unit.

Titles

Have a title at the top of each file. Make sure titles are brief yet relevant to the content contained within.


/**
 * Search Modules
 */

Naming

Syntax

  • Words should be seperated by dashes (e.g. my-new-class), not Camel Case (e.g. myNewClass) or Snake Case (e.g. my_new_class).
  • Names should be based on the B.E.M. (Block Element Modifier) convention. BEM 101 - CSS Tricks

Portability

Avoid naming classes based on specific content. Describe what the class is, instead.

Bad...

.affiliated-logos {}

Better

.slider {}

Don't include elements in selectors.

Bad...

button.btn {}

Better

.btn {}

Picking a Name

  • Select what you want explicitly, rather than relying on circumstance or coincidence. Good Selector Intent will rein in the reach and lead of your styles.
  • Write selectors for resuability, so that you can work more efficiently and reduce waste and repetition.
  • Do not nest selectors unnecessarily, because this will increase specificity and effect where else you can use your styles.
  • Do not qualify selectors unnecessarily, as this will impact the number of different elements you can apply styles to.
  • Keep selectors as short as possible, in order to keep specificity down and performance up.

Declarations

Declaration Order

When writing declarations, try to keep all similar declarations grouped together and separated by new lines from other groups. The suggested order is as follows:

  1. Positioning
  2. Box Model
  3. Visual
  4. Typographic
  5. Misc

.box {
  /* Positioning */
  
  position: absolute;
  top: 0;
  z-index: 100;

  /* Box Model */

  display: block;
  width: 100px;
  float: none;

  /* Visual */

  color: red;
  background-image: url('http://google.com');
  border: 1px solid blue;

  /* Typographic */

  font-size: 1.6rem;
  font-weight: 700;
  text-align: center;

  /* Misc */

  opacity: .5;
  transform: translateY(-50%);
  overflow: auto;
}

Multi-line

Very rarely will you run into an instance where a single line declaration is more appropriate than a multi-line declaration. The only time a single line declaration makes sense is for things such as helper classes.


/**
 * Multi-line
 */

.box {
  width: 100px;
  height: 100px;
}

.bordered {
  border-top: 2px solid white;
  border-bottom: 2px solid white;
}

/**
 * Single Line
 */

.bold { font-weight: 700; }
.hidden { display: none !important; }

Media Queries

Make sure media queries are all placed next to the selectors that they affect and not bundled together at the end of a file.
