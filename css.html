Syntax and Formatting

Titles

Have a title at the top of each file. Make sure titles are brief yet relevant to the content contained within.


/**
 * Search Modules
 */

Multi-line

Very rarely will you run into an instance where a single line declaration is more appropriate than a multi-line declaration. The only time a single line declaration makes sense is for things such as helper classes.


/**
 * Multi-line
 */

.box {
  width: 100px;
  height: 100px;
}

.bordered {
  border-top: 2px solid white;
  border-bottom: 2px solid white;
}

/**
 * Single Line
 */

.bold { font-weight: 700; }
.hidden { display: none !important; }

Naming

Syntax

  • Words should be seperated by dashes (e.g. my-new-class), not Camel Case (e.g. myNewClass) or Snake Case (e.g. my_new_class).
  • Names should be based on the B.E.M. (Block Element Modifier) convention. BEM 101 - CSS Tricks

Portability

Avoid naming classes based on specific content. Describe what the class is, instead.

Bad...

.affiliated-logos {}

Better

.slider {}

Don't include elements in selectors.

Bad...

button.btn {}

Better

.btn {}

Picking a Name

  • Select what you want explicitly, rather than relying on circumstance or coincidence. Good Selector Intent will rein in the reach and lead of your styles.
  • Write selectors for resuability, so that you can work more efficiently and reduce waste and repetition.
  • Do not nest selectors unnecessarily, because this will increase specificity and effect where else you can use your styles.
  • Do not qualify selectors unnecessarily, as this will impact the number of different elements you can apply styles to.
  • Keep selectors as short as possible, in order to keep specificity down and performance up.
